# V1.5 Planning — Airloop

**Scope:** Features that do not require a new architecture — extend V1 without breaking it.
**Timeline:** 6–8 weeks after V1 RC ships to TestFlight.
**Principle:** Engine stays sovereign. Airloop stays template-based. Premium gates get real IAP.

---

## Priority 1 — Push Notifications

**Why first:** The app currently requires the user to open it to know what to do. The anchor time reminder is the single highest-value touch.

| Feature | Description | Rule anchor |
|---------|-------------|-------------|
| Anchor reminder | Push at anchor time if no night logged | R001 |
| Pre-sleep reminder | Push at preSleepStart (90 min before bedtime) | R010 |
| CRP window open | Push at 13:00 when zone is Yellow or Orange | R020 |
| "Log last night" nudge | Push morning after if night not yet logged | New |

**Implementation notes:**
- Use `expo-notifications` (already in Expo SDK 54)
- Schedule notifications locally via `scheduleNotificationAsync` — no backend required
- All times derived from `buildDayPlan()` output — engine remains sovereign
- User opt-in per notification type (pre-sleep, CRP, anchor)
- Respect Do Not Disturb — never push during sleep window

## Priority 2 — In-App Purchase (Premium)

**Why second:** Premium gate UI ships in V1 but the CTA is a placeholder. V1.5 wires the actual purchase.

| Feature | Description |
|---------|-------------|
| IAP product setup | `airloop_premium_monthly` + `airloop_premium_annual` on App Store |
| `expo-in-app-purchases` or `react-native-iap` | Purchase flow + receipt validation |
| Premium state persistence | Store receipt locally; validate on app open |
| Restore purchases | Standard iOS restore flow |

**Gate scope (unchanged from V1):**
- Conflict resolution: 1 free → Premium for unlimited
- Plan recalculation: 1 free → Premium for unlimited

**Notes:**
- Keep client-side validation only in V1.5 (no backend receipt verification)
- Server-side receipt validation is V2 (requires backend)

## Priority 3 — Telemetry (Local)

**Why:** Pattern detection requires data. V1.5 builds the local event log that V2's pattern detector will read.

**What to track (local only, no analytics SDK):**
- App opens (date + time)
- Nights logged (cycles, anchor deviation)
- CRP completions (type, duration, time)
- Premium gate hits (feature, resolved Y/N)
- Chat prompts used (prompt ID only — no content)

**Implementation:**
- `lib/telemetry.ts` — append-only event log in AsyncStorage (`@r90:events:v1`)
- Max 90 days retained, auto-pruned on open
- No network calls — data never leaves the device in V1.5
- Exposed in `exportAllData()` for future user-controlled export

## V2 Scope (Not V1.5)

These require architectural additions beyond what V1.5 can support cleanly:

| Feature | Blocker |
|---------|---------|
| LLM-backed Airloop responses | Requires `llm-bridge.ts`, API key management, latency handling |
| Wearable integration (Apple Health) | Requires `HealthKit` entitlement, data parsing layer |
| Pattern detection | Requires 30+ days telemetry data (V1.5 must ship first) |
| Backend / cloud sync | Requires auth system, API, database — full backend scope |
| Social features | Out of scope for personal recovery tool |
| Server-side receipt validation | Requires backend infrastructure |

## V1 → V1.5 Migration Notes

- `UsageRecord` counts reset on app reinstall (client-side only). V1.5 should tie to IAP receipt, not usage count.
- `@r90:usage:v1` storage key will be deprecated once IAP is wired.
- `PremiumGate` component is already wired — just swap the "Get Premium" CTA to the IAP flow.
